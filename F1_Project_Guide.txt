1. Project Overview & GoalsThis guide provides a detailed, step-by-step implementation plan for the F1 Race Strategy Optimization project. It covers every phase from environment setup through final deployment, including code structure, data pipelines, ML models, simulation engines, and dashboard development.1.1  What You Are BuildingYou will build a modular software platform consisting of five integrated layers:• Data Layer: Automated ingestion and preprocessing of F1 race telemetry from public APIs• Modelling Layer: Machine learning models for lap time prediction and tyre degradation• Simulation Layer: Deterministic race engine + Monte Carlo probabilistic analysis• Optimization Layer: Reinforcement learning agent that recommends pit stop strategies• Interface Layer: REST API + interactive React dashboard1.2  Why This Project MattersIn real Formula 1, strategy engineers process gigabytes of telemetry per race and evaluate thousands of strategy combinations in seconds. This project replicates that system at academic scale, applying genuine Big Data and AI techniques to a real-world, data-rich domain. The result is a portfolio-grade project demonstrating end-to-end data engineering, predictive modelling, simulation, and AI.

Phase 2 — Data Preprocessing & Feature Engineering4.1  Data CleaningCreate src/preprocessing/cleaner.py with the following cleaning steps applied to the raw laps DataFrame:4.1.1  Handling Missing Values• Drop laps where LapTime is null (typically in-laps, out-laps, VSC laps)• Flag pit laps using PitOutTime / PitInTime columns — keep them but mark with is_pit_lap = True• Fill missing TyreLife with forward-fill within each driver stint• Drop laps where Compound is null (unclassified tyre data)4.1.2  Outlier Removal• Remove laps with LapTime > (median + 2.5 * IQR) — these are safety car laps or red flag laps• Remove laps with LapTime < theoretical minimum (1.05 * circuit fastest lap record)• Log all removed laps to a separate CSV for traceability4.1.3  Time Conversion• Convert all timedelta columns (LapTime, SectorTimes) to float seconds using .total_seconds()• Store cleaned data back to data/processed/laps_clean_{circuit}_{year}.parquet4.2  Feature EngineeringCreate src/preprocessing/feature_engineer.py. This step derives the predictive features your models will use:4.2.1  Tyre Features• TyreAge: lap number within the current stint (reset at each pit stop)• TyreCompoundEncoded: encode Soft=0, Medium=1, Hard=2, Inter=3, Wet=4• RelativeTyrePerformance: LapTime - stint mean lap time (measures degradation within stint)• StartingTyreAge: tyre age at the beginning of each stint4.2.2  Race Progression Features• FuelLoad: estimated as starting_fuel - (fuel_burn_rate * LapNumber). Assume ~1.6 kg/lap burn rate, ~105 kg start• FuelEffect: FuelLoad * 0.035 (each kg of fuel adds ~0.035 seconds per lap)• RaceProgress: LapNumber / TotalLaps (0 to 1)• PositionInRace: driver race position per lap (from session position data)4.2.3  Track & Weather Features• TrackEvolution: cumulative rubber laid down — proxy using race lap number (increases over race)• AirTemp, TrackTemp, Humidity: merge from weather DataFrame by timestamp• IsRainingLap: boolean flag from Rainfall column4.2.4  Traffic Features• GapToCarAhead: requires positional data — estimated from sector time deltas between consecutive drivers• IsInTraffic: boolean flag if GapToCarAhead < 1.0 second (DRS window)4.3  Exploratory Data Analysis (EDA)Create notebooks/01_eda.ipynb with the following analysis. 5. Phase 3 — Predictive ModellingThis phase builds three models that form the core predictive engine. Each model is trained independently, evaluated rigorously, and saved as a serialized file for use in the simulation engine.5.1  Model 1: Lap Time Prediction5.1.1  Problem FormulationThis is a supervised regression problem. Given a set of race state features, predict the lap time in seconds for the next lap.5.1.2  Input Features• TyreCompoundEncoded, TyreAge, StartingTyreAge• FuelLoad, FuelEffect• RaceProgress, LapNumber• AirTemp, TrackTemp, Humidity• IsInTraffic, GapToCarAhead• CircuitEncoded (one-hot or label encoded circuit identifier)• DriverEncoded (label encoded driver — captures driver pace baseline)5.1.3  Algorithms to Evaluate• Baseline: Linear Regression — establishes a performance floor• Gradient Boosting: XGBoost or LightGBM — expected best performer for tabular data• Random Forest: good interpretability for feature importance analysis• Neural Network (MLP): 3 hidden layers [256, 128, 64] with ReLU, trained with Adam optimizer5.1.4  Training PipelineCreate src/models/train_laptime.py:7. Load all clean laps from data/processed/8. Apply 80/10/10 train/validation/test split — split by race (not random row split) to avoid data leakage9. Scale numerical features with StandardScaler — fit on training set only10. Train each model, tune hyperparameters with 5-fold cross-validation on the training set11. Evaluate all models on the held-out test set using MAE, RMSE, and R-squared12. Save the best model to src/models/saved/laptime_model.pkl using joblib13. Save the scaler to src/models/saved/scaler.pkl5.1.5  Target PerformanceAim for MAE < 0.5 seconds on clean laps. This means your model predicts lap times within 0.5 seconds on average — acceptable for strategy use.5.2  Model 2: Tyre Degradation Model5.2.1  Problem FormulationModel how tyre performance degrades as TyreAge increases. This will be used to predict lap time at any given tyre age during simulation.5.2.2  Approach• Fit a polynomial regression curve (degree 2 or 3) per compound per circuit• Target variable: lap time delta from fresh tyre baseline (DeltaFromBase)• Input: TyreAge, Compound, CircuitID• The model outputs: how many seconds slower a tyre is at age N vs age 15.2.3  ImplementationCreate src/models/tyre_degradation.py:14. Group laps by Circuit, Compound15. Compute DeltaFromBase = LapTime - min(LapTime in first 3 laps of stint)16. Fit numpy.polyfit(degree=2) on TyreAge vs DeltaFromBase for each group17. Store coefficients in a dictionary keyed by (circuit, compound)18. Save the coefficient dictionary to data/processed/tyre_deg_coefficients.json19. At inference: deg_delta = coeff[0]*age^2 + coeff[1]*age + coeff[2]5.3  Model 3: Pit Stop Time Loss (Pit Delta)5.3.1  What is Pit Delta?Pit delta is the time penalty for making a pit stop — the total time lost from entering the pit lane to resuming full racing speed. This is typically 20–28 seconds depending on the circuit.5.3.2  Implementation20. Extract PitInTime and PitOutTime from session data21. Compute pit_duration = PitOutTime - PitInTime in seconds22. Subtract the time the pit lane car was stationary (actual stop time, ~2.5s)23. Compute pit_delta = pit_duration - expected_race_lap_time_for_that_lap24. Store mean pit delta per circuit in config/pit_deltas.yaml25. This value is used as a constant time penalty in the simulation engine5.4  Model Evaluation & DocumentationCreate notebooks/02_model_evaluation.ipynb documenting:• Learning curves for the lap time model (train vs validation loss by epoch/estimator count)• Residual plot: predicted vs actual lap time scatter plot• Feature importance bar chart (from XGBoost or Random Forest)• Degradation curve plots: fitted curves overlaid on actual data per compound per circuit• Per-circuit model performance breakdown — some circuits are more predictable than others

Phase 4 — Race Simulation EngineThe simulation engine ties together all three models to simulate a full race and compute the expected race time for any given strategy.6.1  Deterministic Race Simulator6.1.1  Core ConceptA strategy is defined as a list of stints. Each stint specifies a starting compound and the lap number at which the driver pits to start the next stint. For example, a two-stop strategy might be: [(Soft, stop at lap 18), (Medium, stop at lap 38), (Hard, finish)].6.1.2  Simulation LoopCreate src/simulation/race_engine.py with a simulate_race() function:26. Input: driver parameters, circuit, strategy (list of stints), total laps, initial position27. Initialize: current_time = 0, current_compound = first compound, tyre_age = 0, lap = 128. For each lap 1 to total_laps:? Compute fuel_load = starting_fuel - fuel_burn_rate * lap? Compute tyre degradation delta from Model 2: deg_delta = deg_model(compound, tyre_age)? Predict base lap time from Model 1 using current features? total_lap_time = base_lap_time + deg_delta + traffic_delta? If this lap is a pit lap: add pit_delta to lap time, reset tyre_age=0, update compound? Accumulate total_time += total_lap_time29. Return total_time — the predicted race time for this strategy6.1.3  Strategy ComparisonWrite a function compare_strategies(strategies_list) that runs simulate_race() for each strategy and returns a sorted DataFrame with total race time, sorted from best to worst. This allows direct comparison of one-stop vs two-stop vs three-stop strategies.6.2  Traffic & Overtaking Model6.2.1  Why Traffic MattersIn a real race, a faster car stuck behind a slower car loses time due to dirty air, inability to push, and potential overtaking attempts. This time loss must be modelled for realistic simulation.6.2.2  Implementation• At each lap, compute the time gap to the car directly ahead based on cumulative race times of all simulated drivers• If gap < 1.0 second: apply traffic_delta = +0.3 to +0.8 seconds (parameterise per circuit based on overtaking difficulty)• If gap > 3.0 seconds: apply traffic_delta = 0.0 (free air)• Use circuit overtaking difficulty ratings from historical data — Monaco is hardest, Bahrain is easier6.3  Safety Car Simulation6.3.1  Safety Car EventsSafety cars (SC) and Virtual Safety Cars (VSC) dramatically impact strategy because teams can pit for free (or cheaply). You must simulate their occurrence probabilistically.6.3.2  Implementation• Historical SC probability per circuit: compute from FastF1 historical data — what fraction of races have had an SC?• Store SC_probability_per_lap per circuit (probability a SC occurs on any given lap)• SC_laps: when SC occurs, all lap times for affected laps become SC_lap_time (fixed at ~1.5x race pace)• Strategic opportunity: if SC occurs while a driver is about to pit naturally, bring forward the pit by 1–2 laps for a free stop• VSC is modelled similarly but with less severe lap time delta (~1.2x race pace)6.4  Monte Carlo Simulation6.4.1  PurposeThe deterministic engine gives one outcome for one set of inputs. Monte Carlo simulation acknowledges uncertainty by running thousands of simulations with varied inputs sampled from probability distributions.6.4.2  Stochastic Variables• Lap time noise: add Gaussian noise N(0, sigma) to each lap time. Sigma estimated from historical variance per circuit.• Pit stop duration: sample from historical pit stop time distribution (Gaussian, mean~2.5s, std~0.3s)• Safety car occurrence: Bernoulli draw on each lap using SC_probability_per_lap• Tyre degradation: add noise to degradation coefficients — sigma estimated from per-stint variance6.4.3  ImplementationCreate src/simulation/monte_carlo.py:30. Input: strategy, n_simulations (default 5000), circuit, driver31. For each simulation iteration: sample all stochastic variables, run simulate_race(), record total_time32. Output: distribution of race times (list of n_simulations values)33. Compute statistics: mean, median, std, P10, P90 (10th and 90th percentiles)34. Compare two strategies: strategy A wins in X% of simulationsVisualise results as overlapping histograms — one distribution per strategy. The area of overlap shows uncertainty in which strategy wins.
7. Phase 5 — Reinforcement Learning Strategy AgentThe RL agent learns to make pit stop decisions dynamically during a simulated race, rather than following a fixed pre-planned strategy. It adapts to changing conditions such as safety cars, tyre degradation pace, and traffic.7.1  Reinforcement Learning Framework7.1.1  Algorithm ChoiceUse Proximal Policy Optimization (PPO) from the stable-baselines3 library. PPO is well-suited for discrete action spaces and continuous state spaces, is stable to train, and has strong community support.7.1.2  Why RL for This Problem?Traditional optimization finds the best strategy before the race starts. RL learns a policy — a mapping from race state to action — that can make decisions reactively during the race. If a safety car appears unexpectedly, the RL agent can adjust, while a pre-planned strategy cannot.7.2  Custom Gym Environment7.2.1  Creating the EnvironmentCreate src/rl_agent/f1_env.py inheriting from gymnasium.Env:7.2.2  State Space (Observation)The state is a vector of 12 numerical values representing the current race situation:• current_lap (normalized 0–1)• tyre_age (normalized by max expected stint length)• compound_encoded (0–4)• current_position (1–20, normalized)• gap_to_leader (seconds)• gap_to_car_ahead (seconds)• estimated_tyre_degradation_rate (seconds/lap)• fuel_load_remaining (normalized)• track_temp (normalized)• safety_car_active (binary 0/1)• laps_remaining (normalized 0–1)• pit_stops_made (0, 1, 2, 3)7.2.3  Action SpaceDiscrete action space with the following actions available each lap:• Action 0: Stay out (do not pit)• Action 1: Pit for Soft tyres• Action 2: Pit for Medium tyres• Action 3: Pit for Hard tyres• Action 4: Pit for Intermediate tyres (if wet)7.2.4  Reward FunctionThe reward function is critical — it defines what the agent optimizes for. Design it carefully:• Per-lap reward: small positive reward for gaining positions (+0.5 per position gained)• Per-lap penalty: small negative reward for losing positions (-0.5 per position lost)• Pit stop reward: penalty for total time lost in pit (-pit_delta / 100)• Final reward: large reward based on finishing position. Position 1 = +100, Position 2 = +80, ..., Position 20 = +5• Illegal action penalty: if agent pits on a lap where it already pitted, return -50 reward and end episode7.2.5  Episode FlowEach episode simulates one full race from lap 1 to final lap. The environment calls the simulator internally, stepping through laps and returning observations, rewards, and done signals at each step.7.3  Training the RL AgentCreate src/rl_agent/train_agent.py:35. Instantiate F1RaceEnv — wrap with DummyVecEnv for stable-baselines3 compatibility36. Create PPO model: policy='MlpPolicy', learning_rate=3e-4, n_steps=2048, batch_size=64, n_epochs=1037. Train for 500,000 timesteps: model.learn(total_timesteps=500_000)38. Evaluate every 50,000 steps using EvalCallback — save best model39. Log training metrics (mean reward, episode length) to tensorboard40. Save final model to src/rl_agent/saved/f1_ppo_agent.zipTraining tip: randomize circuit, starting position, and initial strategy in each episode's reset() to force the agent to generalize rather than memorize one race.7.4  Evaluating the RL AgentAfter training, run 200 evaluation episodes with deterministic=True (no exploration). Compare:• Average finishing position: RL agent vs. best static strategy from Monte Carlo• Distribution of race outcomes: RL agent should have lower variance (adapts to safety cars)• Strategy profiles: visualize what tyre compounds the agent chooses and when it pits under different conditions

11. Common Pitfalls & How to Avoid Them11.1  Data Pitfalls• Data leakage: Never split data randomly by row. Split by race/season to ensure test races are completely unseen during training.• Out-lap / in-lap contamination: Always remove pit entry and exit laps before training lap time models — these are artificially slow and will skew predictions.• Tyre compound inconsistency: FastF1 sometimes returns compound names like 'SOFT' and other times '1' (compound number). Normalize all compound names at ingestion.• Missing weather data: Not all sessions have weather data. Build fallback logic to use circuit average temperatures when weather is unavailable.11.2  Modelling Pitfalls• Overfitting to one circuit: Train on multiple circuits and evaluate on a held-out circuit. A model that only knows Bahrain will fail on Monaco.• Feature scaling: Always fit scaler on training data only. Applying scaler fitted on full dataset is a subtle form of data leakage.• Tyre age extrapolation: Your degradation model should refuse to predict for tyre ages beyond the training distribution. Add a max_tyre_age guard.11.3  Simulation Pitfalls• Strategy space explosion: Testing all possible strategies is computationally infeasible. Limit to logical strategies: 1-stop, 2-stop, 3-stop with reasonable compound choices.• Circular dependency in traffic model: Position depends on lap time which depends on traffic which depends on position. Resolve by using previous lap positions to determine current traffic.11.4  RL Pitfalls• Sparse rewards: If the agent only gets reward at the end, it struggles to learn. Add intermediate position-gain rewards to provide more frequent learning signal.• Reward scale imbalance: If final position reward (100) dwarfs per-lap rewards (0.5), the agent ignores per-lap signals. Carefully balance reward magnitudes.• Non-stationarity: Randomize the starting conditions in each episode reset so the agent cannot memorize a fixed race scenario.
